# WPM Visual Baselines

Full-page visual-regression **baselines** for the Blaze Commerce WPM (WordPress Maintenance) fleet. One folder per site; each holds desktop / tablet / mobile screenshots of the key pages, plus a summary gallery.

These are the *known-good* screenshots captured **before** a round of updates. After updates we re-shoot and pixel-diff against them to catch CSS/layout regressions.

## How to see what changed

This repo is structured so differences are easy to spot **two ways**:

1. **GitHub image diff (per file).** Every screenshot lives at a *stable path* (`<site>/<viewport>/<page>.png`). When a baseline is refreshed, the PNG at that path changes — open the commit/PR on GitHub and click the image to get the **2-up / swipe / onion-skin** comparison built into GitHub.
2. **Git history (timeline).** `git log -- <site>/<viewport>/<page>.png` shows every prior version of that exact shot over time — a visual time machine across months of WPMs.

For a fast eyeball of a whole site, open that site's **`index.html`** — one row per page, three columns (desktop / tablet / mobile).

## Structure

```
<site-slug>/
├── index.html              ← full-page summary gallery (desktop · tablet · mobile)
├── desktop/<page>.png       (1440×900)
├── tablet/<page>.png        (768×1024)
└── mobile/<page>.png        (375×812)
```

## How baselines are produced

Generated by the BackstopJS engine in the WPM repo: `wpm/scripts/visual-regression/`
(`misMatchThreshold 0.05`, `requireSameDimensions true`, dynamic-content masked, lazy-load + animations frozen for determinism).

```bash
# in wpm/scripts/visual-regression — BEFORE updates:
./vr-run.sh <slug> <staging-url> reference
# publish into this repo (stable paths + gallery):
node vr-publish.mjs --slug <slug> --run ~/bc-visual-regression/<slug> --dest <this-repo>/<slug>
# commit + push → the new baseline is the reference for next time
```

Refreshing a baseline overwrites the same paths, so the commit *is* the diff. Capture baseline and post-update test on the **same environment** (both staging) — comparing live vs staging produces false diffs.
